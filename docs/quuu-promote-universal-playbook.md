# Quuu Promote 범용 콘텐츠 플레이북

최종 확인일: 2026-07-23  
용도: 어떤 웹사이트 프로젝트에서도 재사용할 수 있는 Quuu Promote용 아티클 기획, 제작, 발행, 검증, 제출 절차

이 문서는 WePhoto에서 만든 범용 기준을 기반으로 하며, OZSNAP에서 실제 영어 아티클 두 편을 기획하고 Cloudflare Pages에 배포하면서 확인한 콘텐츠, 이미지, 인포그래픽, 캐시, 라이브 검증 절차를 합친 실행판이다.

## 1. 목표

Quuu Promote에 올릴 글은 승인만 받는 글이 아니라 다음 세 가지 역할을 동시에 수행해야 한다.

1. 독자가 공유할 만큼 유용한 독립형 콘텐츠여야 한다.
2. 검색엔진과 AI 검색 시스템이 내용을 이해하고 인용하기 쉬워야 한다.
3. 유입된 독자가 사이트의 관련 가이드, 도구, 제품 이해 페이지로 자연스럽게 이동해야 한다.

권장 흐름은 다음과 같다.

`Quuu 공유 → 독립형 아티클 방문 → 본문에서 문제 해결 → 관련 내부 페이지 사용 → 추가 탐색 또는 전환`

글이 지나치게 홍보성으로 보이면 Quuu 승인 가능성이 떨어진다. 반대로 내부 연결이 전혀 없으면 트래픽이 한 페이지에서 끝난다. 핵심은 **도움이 먼저이고, 내부 이동은 그 도움을 완성하는 다음 단계**가 되게 만드는 것이다.

## 2. Quuu 공식 기준에서 지켜야 할 것

아래 항목은 [Quuu Promote 공식 품질 기준](https://quuu.co/blog/quuu-promote-quality-criteria)을 프로젝트 시작 시 다시 확인한다. 플랫폼 정책은 변경될 수 있다.

- 홈페이지, 카테고리 목록, 인덱스 페이지가 아닌 개별 콘텐츠 URL을 제출한다.
- 영어로 작성한다.
- 판매 페이지, 광고, 출시 공지, 특정 제품 홍보문처럼 보이지 않게 한다.
- 로그인, 이메일 입력, 결제 없이 본문 전체를 읽을 수 있어야 한다.
- Open Graph 대표 이미지가 정상 작동해야 한다.
- 최소 프로모션 기간 동안 낡지 않을 evergreen 주제를 우선한다.
- 문법, 철자, 문장 흐름을 교정한다.
- 깨진 링크나 본문과 무관한 링크를 남기지 않는다.
- 판매 페이지로 향하는 링크를 과도하게 넣지 않는다.
- 특정 지역이나 아주 좁은 독자만 이해할 주제를 넓은 카테고리에 제출하지 않는다.
- 강한 정치적, 논쟁적, 편향적 표현을 피한다.
- 복제하거나 얕게 바꾼 글이 아니라 독창적인 설명, 구조, 예시를 제공한다.
- 같은 URL은 한 번에 한 카테고리에서만 프로모션한다.

Quuu는 독자를 실질적으로 돕거나, 알리거나, 즐겁게 하거나, 영감을 주는 콘텐츠를 선호한다. 체크리스트, 계산 방법, 비교표, 작업 순서, 템플릿처럼 읽고 바로 사용할 수 있는 결과물이 유리하다.

## 3. 프로젝트 입력 시트

글을 쓰기 전에 아래 항목을 채운다. 답이 없으면 콘텐츠가 좋아도 사이트에 어떤 도움을 줄지 결정하기 어렵다.

| 항목 | 입력 내용 |

| 사이트/브랜드 | `[브랜드명과 도메인]` |

| 사이트가 제공하는 가치 | `[가이드, 도구, 서비스, 데이터 등]` |
| 이번 글의 한 가지 문제 | `[독자가 검색하거나 공유할 질문]` |
| 핵심 검색 의도 | `[how-to, 비교, 체크리스트, 설명 등]` |
| 연결할 내부 페이지 | `[관련 가이드/도구/허브 3~5개]` |
| 신뢰할 외부 출처 | `[공식 문서 또는 1차 출처 2~5개]` |
| 적합한 Quuu 카테고리 | `[가장 좁고 정확한 주제]` |
| 성공 지표 | `[승인, 공유, 클릭, 내부 이동, 전환]` |

## 4. 주제 선정 기준

후보 주제마다 0~2점을 주고 총점이 높은 것을 우선한다.

| 평가 항목 | 0점 | 1점 | 2점 |
| --- | --- | --- | --- |
| 즉시 활용성 | 읽고 끝남 | 아이디어 제공 | 바로 실행 가능한 결과물 |
| 독자 범위 | 지나치게 좁음 | 특정 업계 다수 | 카테고리 독자에게 폭넓게 유용 |
| 지속성 | 곧 낡음 | 일부 업데이트 필요 | 장기간 유효 |
| 사이트 연결성 | 관련 페이지 없음 | 약한 관련성 | 자연스러운 다음 단계 3개 이상 |
| 독창성 | 흔한 요약 | 새로운 구조 | 고유 프레임워크, 예시 또는 데이터 |
| 출처 품질 | 출처 없음 | 2차 출처 | 공식 문서와 1차 출처 |

권장 내부 기준은 9점 이상이다. 점수보다 중요한 것은 다음 질문이다.

> 이 글만 읽어도 문제가 상당 부분 해결되고, 더 실행하려는 독자에게 우리 사이트의 다른 페이지가 실제로 도움이 되는가?

## 5. 사이트에 도움이 되는 내부 링크 설계

### 5.1 링크 목적지 우선순위

1. 글의 한 단계를 직접 수행하는 무료 도구
2. 예시나 선택지를 보여주는 상세 가이드
3. 다음 문제를 해결하는 관련 아티클
4. 주제별 허브 또는 카테고리 페이지
5. 서비스/제품 페이지는 꼭 필요한 경우에만 보조 링크로 사용

홈페이지 링크만 반복하면 독자가 다음에 무엇을 해야 할지 알기 어렵다. 가능한 한 구체적이고 유용한 내부 페이지로 연결한다.

### 5.2 배치 원칙

- 본문 중간에 문맥형 내부 링크 3~5개를 둔다.
- 각 링크는 바로 앞에서 설명한 작업을 실행하거나 확장해야 한다.
- 글 하단에는 `Keep going`, `Next step`, `Related resources` 같은 관련 링크 묶음을 둔다.
- 같은 목적지를 같은 앵커 텍스트로 반복하지 않는다.
- 과도한 정확 일치 키워드 대신 독자가 클릭 결과를 예상할 수 있는 자연스러운 문장을 쓴다.
- 내부 링크를 광고 배너처럼 만들지 않는다.
- 링크가 없어도 문단 자체는 완전한 정보를 제공해야 한다.

### 5.3 문맥형 링크 템플릿

```text
[작업 단계 이름]
[현재 단계에서 자주 생기는 문제와 짧은 해결 설명]
[도구/가이드로 이동하는 구체적인 링크 문구]
```

예시:

```text
Check the crop before exporting
Enter the master dimensions and target format to see how much needs to be cropped.
Use the image-size calculator →
```

좋은 링크 문구:

- `Compare the available options`
- `Calculate the result before exporting`
- `Open the complete 12-step checklist`
- `Check the current requirements`

피할 링크 문구:

- `Click here`
- `Buy now`
- `Our amazing service`
- 키워드를 부자연스럽게 반복한 문구

## 6. 권장 아티클 구조

Quuu가 공식적인 최소 글자 수를 제시한 것으로 가정하지 않는다. 아래 분량은 승인 규칙이 아니라 충분한 실용성을 확보하기 위한 내부 권장치다.

### 기본 사양

- 권장 분량: 영어 약 1,200~2,200단어
- H1: 독자가 얻을 결과와 조건을 명확하게 표시
- 도입부: 문제, 결과, 이 글에서 제공할 방법을 2~3문단으로 설명
- H2 섹션: 7~14개
- 원본 또는 사용 권한이 있는 이미지: 2~4개
- 문맥형 내부 링크: 3~5개
- 신뢰할 외부 출처: 2~5개
- 마지막 실행 체크리스트 또는 요약 포함

### 재사용 가능한 개요

```markdown
# How to [achieve outcome] in/without [constraint]

[독자가 겪는 문제]
[이 글에서 얻을 구체적인 결과]
[방법 전체를 한 문단으로 요약]

## Define the outcome
[성공 조건과 범위]

## Prepare the minimum inputs
[준비물, 데이터, 조건]

## Use this framework
[고유한 단계, 표, 매트릭스 또는 판단 기준]

## Follow the process
[번호가 있는 실행 순서]

## Avoid the common mistakes
[문제, 원인, 수정법]

## Put the result into practice
[첫 번째 문맥형 내부 링크]

## Adapt it for different situations
[여러 독자에게 유용하도록 변형 사례]

## Check the final result
[품질 검증법과 두 번째 내부 링크]

## Copy this checklist
[독자가 바로 복사할 수 있는 체크리스트]

## Sources and further reading
[공식/1차 출처]

## Keep going
[관련 내부 페이지 3~5개]
```

## 7. 글쓰기 품질 기준

- 첫 문단에서 질문에 직접 답한다.
- 각 섹션 첫 문장은 해당 섹션의 결론이 되게 한다.
- 한 문단에는 한 가지 핵심만 담는다.
- 추상적인 조언 뒤에는 수치, 예시, 단계 또는 판단 기준을 붙인다.
- 독자가 복사할 수 있는 표, 체크리스트, 순서표 중 하나 이상을 제공한다.
- 브랜드 이름은 필요한 곳에서만 사용한다.
- 검증할 수 없는 성과 수치나 과장 표현을 만들지 않는다.
- 전문 용어는 처음 등장할 때 짧게 설명한다.
- 인용이 필요한 사실은 공식 문서나 원자료에 연결한다.
- 지역 정보가 핵심이 아니라면 특정 지역에만 적용되는 표현을 일반화한다.

AI 검색과 인용 가능성을 높이려면 각 핵심 답변이 주변 문맥 없이도 이해되는 짧은 독립 블록이 되게 한다. 제목과 H2는 사람들이 실제로 묻는 질문에 가깝게 작성하고, 게시일 또는 수정일과 작성 주체를 명확히 표시한다.

## 8. 이미지 제작 기준

### 권장 구성

1. 대표 이미지: 글의 전체 작업이나 결과를 보여주는 3:2 이미지
2. 방법 이미지: 프레임워크, 작업 환경 또는 프로세스
3. 결과 이미지: 완성 예시, 비교 또는 여러 결과의 모음

### 기술 기준

- 원본 또는 적법하게 사용 가능한 이미지
- 가능한 경우 WebP 또는 AVIF
- 대표 이미지 권장 비율: 3:2 또는 사이트의 기존 OG 규격
- 명시적 `width`와 `height`
- 내용을 설명하는 자연스러운 alt 텍스트
- 이미지 안에 긴 문장, 워터마크, 의미 없는 장식 텍스트를 넣지 않음
- 모바일에서도 핵심 피사체가 잘리지 않는지 확인
- Open Graph 이미지 URL이 외부에서 200으로 열리는지 확인

## 9. 발행 전 SEO 및 기술 점검

### 페이지 자체

- [ ] 고유한 `<title>`
- [ ] 핵심 결과가 포함된 meta description
- [ ] H1 한 개
- [ ] 논리적인 H2/H3 구조
- [ ] 자기 자신을 가리키는 canonical URL
- [ ] `Article` 또는 `BlogPosting` 구조화 데이터
- [ ] 게시일, 수정일, 작성자/발행자 표시
- [ ] 대표 OG 이미지와 소셜 제목/설명
- [ ] 모바일 가로 넘침 없음
- [ ] 콘솔 오류 없음
- [ ] 주요 내용이 서버 렌더링 또는 prerender HTML에 존재

### 검색·크롤링

- [ ] HTTP 200이며 불필요한 리다이렉트 없음
- [ ] `noindex`가 없음
- [ ] sitemap에 canonical URL 포함
- [ ] robots.txt가 일반 검색 크롤러를 막지 않음
- [ ] 조직 정책에 맞춰 ChatGPT-User, PerplexityBot, Bingbot 등 검색·참조용 크롤러 접근 검토

### 링크

- [ ] 내부 링크 목적지가 모두 200
- [ ] 외부 출처가 모두 열림
- [ ] 리다이렉트 대상이 의도한 최종 페이지인지 확인
- [ ] 외부 새 창 링크에는 `rel="noopener noreferrer"` 적용
- [ ] 판매 페이지 링크가 과도하지 않음
- [ ] 앵커 텍스트가 목적지 내용을 정확히 설명

## 10. 카테고리 선택법

가장 큰 카테고리가 아니라 가장 관련성이 높은 카테고리를 고른다. Quuu의 [Smart Blogger 사례](https://quuu.co/blog/case-study-how-smart-blogger-overcame-ad-blindness-using-quuu-promote)도 단순 도달 규모보다 올바른 독자와의 주제 일치를 강조한다.

선택 순서:

1. 글의 핵심 결과를 한 문장으로 쓴다.
2. 그 결과가 필요한 사람의 직업이나 관심사를 적는다.
3. 화면에 표시된 카테고리 중 해당 독자가 기대할 가장 구체적인 항목을 고른다.
4. 세부 주제가 있다면 넓은 상위 카테고리보다 정확한 세부 주제를 우선한다.
5. 적합한 카테고리가 없으면 억지로 제출하지 말고 글의 범위 또는 제출 채널을 재검토한다.

예시:

| 글의 핵심 | 1순위 | 2순위 |
| --- | --- | --- |
| 소셜 콘텐츠 계획 | Social Media | Content Marketing |
| 블로그 작성 방법 | Writing | Content Marketing |
| 검색 최적화 | SEO | Digital Marketing |
| 창업 세금 체크리스트 | 해당 국가의 Finance/Tax | Entrepreneurship |
| 원격 팀 운영 | Remote Working | Leadership |

Quuu의 다중 브랜드 안내에서는 Marketing 아래 세부 주제로 Social Media Strategy, Content Marketing, Email Marketing 등을 예로 든다. 실제 제출 화면의 명칭이 다르면 표시된 선택지 중 가장 가까운 것을 사용한다.

## 11. 제출 패키지 템플릿

```markdown
### Quuu Promote Submission

- Project/brand: [브랜드]
- Live article URL: https://example.com/guides/article-slug
- Canonical URL: https://example.com/guides/article-slug
- HTTP status: 200
- Category: [가장 정확한 카테고리]
- Secondary category if unavailable: [차선]
- Article title: [실제 H1]
- Share text A: [결과와 실용성을 강조한 자연스러운 문장]
- Share text B: [질문 또는 문제에서 시작하는 대안 문장]
- OG image URL: https://example.com/assets/article-hero.webp
- Main audience: [독자]
- Tangible value: [체크리스트/계산법/프레임워크/템플릿]
- Internal next steps: [내부 페이지 3~5개]
- External sources checked: [날짜]
- Live QA checked: [날짜와 담당자]
- Submission status: Prepared / Submitted / Approved / Rejected / Finished
- Submission date: [YYYY-MM-DD]
- Promotion end date: [YYYY-MM-DD 또는 미정]
- Notes/rejection reason: [있는 경우 원문 그대로]
```

### 공유 문구 작성법

공유 문구는 글의 결과를 약속하되 광고처럼 쓰지 않는다.

```text
Plan [valuable outcome] with this practical [framework/checklist/process]. It covers [two or three concrete benefits] without [common frustration].
```

```text
Trying to [solve problem]? This guide shows how to [specific result], including [useful component] and [useful component].
```

피해야 할 표현:

- `The ultimate revolutionary solution`
- `Buy`, `Book`, `Limited offer`
- 검증할 수 없는 `best`, `guaranteed`, `number one`
- 글에 없는 결과를 약속하는 클릭 유도 문구

## 12. 라이브 검증 절차

로컬 빌드 성공만으로 제출하지 않는다. 배포 후 canonical URL을 직접 확인한다.

```text
1. 배포 상태가 success인지 확인
2. 제출 URL을 새 세션에서 열어 HTTP 200 확인
3. 리다이렉트가 없는지 확인
4. canonical이 제출 URL과 동일한지 확인
5. 제목, 본문, 이미지가 실제로 보이는지 확인
6. 모든 내부/외부 링크의 최종 목적지 확인
7. OG 이미지 URL을 직접 열어 200 확인
8. 데스크톱과 모바일에서 가로 넘침, 팝업, 콘솔 오류 확인
9. sitemap에서 URL 확인
10. 검증 시간과 결과를 상태 문서에 기록
```

간단한 HTTP 확인 예시:

```bash
curl -I https://example.com/guides/article-slug
curl -I https://example.com/assets/article-hero.webp
```

페이지 소스에서 확인할 항목:

```html
<link rel="canonical" href="https://example.com/guides/article-slug">
<meta property="og:image" content="https://example.com/assets/article-hero.webp">
<script type="application/ld+json">...</script>
```

## 13. 승인 전 최종 체크리스트

- [ ] 제출 대상이 홈페이지가 아닌 독립형 아티클이다.
- [ ] 영어이며 누구나 로그인 없이 읽을 수 있다.
- [ ] 최소 한 달 이상 유효한 주제다.
- [ ] 독자가 즉시 사용할 체크리스트, 표, 단계 또는 템플릿이 있다.
- [ ] 특정 상품을 팔기 위한 글처럼 보이지 않는다.
- [ ] 본문 내부 링크 3~5개가 맥락에 맞는다.
- [ ] 내부 링크는 독자의 다음 작업을 실제로 돕는다.
- [ ] 공식 또는 1차 외부 출처를 사용했다.
- [ ] 깨진 링크가 없다.
- [ ] 대표 이미지와 OG 이미지가 정상이다.
- [ ] canonical, sitemap, 구조화 데이터가 정상이다.
- [ ] 카테고리가 글의 실제 독자와 정확히 맞는다.
- [ ] 공유 문구가 전문적이지만 판매 문구처럼 들리지 않는다.
- [ ] 라이브 URL, 이미지, 링크를 배포 후 다시 확인했다.
- [ ] 제출 상태를 `Prepared`와 `Submitted`로 구분해 기록한다.

## 14. 제출 후 측정

최소한 다음을 기록한다.

| 지표 | 목적 |
| --- | --- |
| 승인/거절 상태 | 콘텐츠 적합성 확인 |
| 거절 사유 원문 | 다음 제출 개선 |
| Quuu 공유 수 | 배포 범위 파악 |
| 아티클 방문 수 | 실제 유입 확인 |
| 내부 링크 클릭률 | 사이트 내 이동 효과 확인 |
| 도구/가이드 도달 수 | 핵심 자산 기여도 확인 |
| 참여 시간과 스크롤 | 콘텐츠 품질 확인 |
| 전환 또는 문의 | 최종 비즈니스 효과 확인 |

분석 도구를 사용한다면 Quuu 유입과 문맥형 내부 링크 클릭을 구분해 이벤트를 설정한다. 결과가 낮을 때는 링크 수를 늘리기 전에 카테고리, 공유 문구, 첫 화면의 가치 제안, 내부 링크 위치를 차례로 점검한다.

## 15. 재사용 실행 프롬프트

아래 내용을 새 프로젝트에서 복사해 사용할 수 있다.

```text
이 프로젝트에서 Quuu Promote에 제출할 영어 독립형 아티클 하나를 만들어 주세요.

목표:
- Quuu 승인 기준을 충족하는 비판매성 evergreen 콘텐츠
- 독자가 바로 사용할 수 있는 구체적인 프레임워크, 예시, 체크리스트 제공
- 사이트의 핵심 가이드나 무료 도구로 자연스럽게 이어지는 문맥형 내부 링크 3~5개
- 공식 또는 1차 출처 2~5개
- 원본 또는 적법한 대표 이미지와 본문 이미지 2~4개

진행 순서:
1. 프로젝트의 독자, 핵심 가치, 기존 페이지와 도구를 파악한다.
2. 후보 주제를 평가하고 가장 유용하면서 사이트 연결성이 높은 하나를 고른다.
3. 글은 영어 약 1,200~2,200단어를 내부 권장 범위로 삼되 내용에 맞게 조정한다.
4. 홍보 문구 대신 실행 가능한 설명을 쓴다.
5. 본문 중 필요한 작업 단계에서만 내부 링크를 넣고 하단에 관련 링크 묶음을 둔다.
6. title, description, canonical, OG image, Article schema, sitemap을 적용한다.
7. lint/build와 배포를 확인하고 라이브 URL, 이미지, 모든 링크의 HTTP 응답을 검사한다.
8. Quuu 제출용 URL, 추천 카테고리, 공유 문구 2개, 검증 결과를 Markdown으로 남긴다.

주의:
- 홈페이지나 카테고리 인덱스를 제출하지 않는다.
- 글을 특정 제품 광고처럼 만들지 않는다.
- 깨진 링크, 과도한 판매 링크, 키워드 반복을 피한다.
- 준비 완료와 실제 제출 완료를 구분해 보고한다.
```

## 16. 짧은 적용 예시

WePhoto에서는 `한 번의 90분 촬영으로 30일치 소셜 콘텐츠를 계획하는 방법`을 독립형 아티클로 만들었다. 글 자체에는 촬영 매트릭스, 시간표, 게시물 배분, 체크리스트를 제공하고, 작업 단계에 맞춰 포즈북, Reels 안전영역 도구, 캡션 카운터, 이미지 크기 계산기로 연결했다.

이 예시에서 재사용할 핵심은 사진이라는 주제가 아니다. **독립형 문제 해결 글 안에서 사이트의 핵심 자산이 해결 과정의 도구가 되게 만든 구조**다.

## 17. 시각 자료 운영 원칙

대표 이미지와 본문 설명 자료는 목적이 다르다. 한 장으로 모든 역할을 해결하려 하지 않는다.

### 17.1 첫 화면 커버

커버의 역할은 글의 결과와 감정을 즉시 전달해 클릭 후 이탈을 줄이는 것이다.

- 문제의 우울한 장면보다 독자가 얻고 싶은 결과를 우선한다.
- 사람 중심 주제라면 얼굴, 표정, 시선과 관계가 모바일 크롭에서도 남아야 한다.
- 밝기와 색 대비가 낮은 이미지는 실제 페이지 첫 화면에서 다시 평가한다.
- 직접적인 눈맞춤은 주목도를 높일 수 있지만 글의 주제와 어긋나는 자극만 만들면 안 된다.
- 대표 이미지 교체 요청이면 본문 비교 이미지와 설명 그래픽은 별도 요청이 없는 한 유지한다.
- 생성형 이미지에는 실제 인물이나 실제 결과로 오해하게 할 문구를 붙이지 않는다.

### 17.2 본문 설명 자료

본문 자료는 독자가 내용을 더 빨리 이해하거나 기억하게 해야 한다.

| 목적 | 권장 형식 |
| --- | --- |
| 조건별 결과 비교 | 동일 구도의 3~4패널 비교 이미지 |
| 단계와 판단 흐름 | HTML/CSS 인포그래픽 또는 순서도 |
| 실제 환경과 장비 | 현장 사진 |
| 수치와 기준 | 표, 데이터 카드, 타임라인 |
| 실행 요약 | 복사 가능한 체크리스트 |

HTML/CSS 인포그래픽은 이미지 속 작은 글자보다 모바일 가독성, 접근성, 검색 이해도가 좋다. 데스크톱 4열, 태블릿 2열, 모바일 2열 또는 1열로 실제 화면에서 높이와 글자 잘림을 확인한다.

### 17.3 이미지 파일 세트

대표 이미지는 최소한 다음 세트를 만든다.

```text
article-hero.webp          원본 비율, 본문/데스크톱
article-hero-960.webp      반응형 전송용
article-og.jpg             1200 x 630 소셜 공유용
```

- `srcset`, `sizes`, `width`, `height`를 지정한다.
- OG 이미지는 본문 크롭과 별도로 얼굴과 핵심 피사체가 남는지 확인한다.
- 대표 이미지를 크게 바꿀 때는 새 파일명을 사용한다. 같은 URL을 덮어쓰면 CDN, 브라우저, Facebook/X/LinkedIn 미리보기에 이전 이미지가 남을 수 있다.
- 파일명 예: `article-hero-v2.webp`, `article-og-v2.jpg` 또는 의미가 달라진 새 이름.
- 사용하지 않는 이전 파일은 전체 참조 검색 후 제거한다.

## 18. 처음부터 라이브까지의 표준 실행 순서

### Phase A: 사이트와 독자 파악

1. 프로젝트 메모리와 현재 상태 문서를 읽는다.
2. canonical 페이지, sitemap, robots, 기존 콘텐츠와 핵심 도구를 조사한다.
3. 외부 링크와 내부 링크 목적지를 실제 응답으로 확인한다.
4. 사이트가 판매하려는 것보다 독자가 해결하려는 문제를 먼저 정의한다.

### Phase B: 주제와 카테고리 결정

1. 후보 주제를 즉시 활용성, 지속성, 독창성, 사이트 연결성으로 점수화한다.
2. 지역 사례가 있어도 범용 독자가 사용할 수 있는 문제 해결 구조로 넓힌다.
3. 카테고리는 가장 큰 항목이 아니라 글의 결과를 기대할 가장 정확한 독자 항목을 고른다.
4. 실제 제출 화면의 카테고리 명칭을 기준으로 최종 확정한다.

### Phase C: Markdown 제작 패키지

코드보다 먼저 또는 동시에 다음 내용을 하나의 전용 Markdown에 저장한다.

- SEO title, description, canonical, H1
- 대상 독자와 콘텐츠 유형
- 1순위/차선 카테고리
- 완성 원고
- 이미지 URL, alt, 캡션, OG URL
- 내부 링크와 외부 출처
- 공유 문구 2~3개
- 제출 체크리스트와 상태

이 파일은 원고, 제출 화면 입력값, 향후 수정 이력을 함께 보존하는 소스다.

### Phase D: 사이트 구현

1. 독립 URL과 canonical을 만든다.
2. title, description, OG, Article/BlogPosting schema를 적용한다.
3. 사이트 기존 헤더, 메뉴, 푸터와 연결한다.
4. 본문 내부 링크를 문맥에 맞게 배치한다.
5. sitemap과 홈페이지 또는 관련 허브에서 발견 가능한 링크를 추가한다.
6. CSS/JS를 변경했으면 minified 파일과 캐시 버전을 함께 갱신한다.

### Phase E: 로컬 QA

1. HTML lint와 빌드 검사를 실행한다.
2. `git diff --check`를 실행한다.
3. 데스크톱, 태블릿, 375px 모바일을 확인한다.
4. 가로 넘침, 글자 겹침, 이미지 크롭, lazy image 로딩을 확인한다.
5. 콘솔 오류와 링크 목적지를 확인한다.
6. OG 이미지 1200 x 630 크롭을 직접 본다.

### Phase F: 릴리스와 라이브 QA

1. 의도한 파일만 stage한다.
2. 커밋하고 원격 기본 브랜치에 push한다.
3. Cloudflare 또는 배포 플랫폼의 전파를 기다린다.
4. 새 경로가 잠시 404일 수 있으므로 고유 본문 마커가 나타날 때까지 poll한다.
5. 라이브 URL, 이미지 URL, sitemap, robots를 `200`으로 확인한다.
6. 라이브 DOM에서 canonical, OG, schema, 새 CSS 버전과 콘텐츠 마커를 확인한다.
7. 라이브 모바일 브라우저에서 overflow와 콘솔 오류를 다시 확인한다.
8. 상태 문서에 커밋, URL, 검증 결과를 기록한다.

### Phase G: Quuu 제출

사이트 발행과 Quuu 제출은 별도 작업이다. 사용자가 콘텐츠 제작까지만 요청하면 Quuu 계정에 제출하지 않는다.

상태는 다음 용어로 구분한다.

```text
Draft       원고 작성 중
Prepared    제출 패키지와 라이브 페이지 준비 완료
Published   사이트에는 발행됨
Submitted   Quuu 제출 화면에서 실제 전송 완료
Approved    Quuu 검수 승인 확인
Rejected    거절 및 사유 확인
Finished    프로모션 기간 종료
```

`Published, not submitted`처럼 두 시스템의 상태를 함께 적으면 오해가 적다.

## 19. 카테고리 선택 실전 규칙

카테고리는 글 제목의 단어가 아니라 **그 글을 공유할 사람의 관심사**를 기준으로 선택한다.

예를 들어 날씨를 읽어 야외 사진 촬영을 계획하는 글은 Weather보다 Photography가 정확하다. 선택지가 있다면 다음처럼 좁은 항목을 우선한다.

```text
Photography Tips
Photography
Outdoor Photography
Travel Photography
```

이 순서는 고정된 Quuu 카테고리 목록이 아니라 해당 글에 대한 우선순위 예시다. 플랫폼 선택지는 바뀔 수 있으므로 제출 당일 화면을 다시 확인한다.

카테고리 결정 질문:

1. 이 글을 읽고 바로 행동할 사람은 누구인가?
2. 그 사람이 Quuu에서 구독할 가장 구체적인 관심사는 무엇인가?
3. 글의 70% 이상이 그 관심사에 직접 유용한가?
4. 지역명이나 보조 소재 때문에 잘못된 넓은 카테고리를 고른 것은 아닌가?
5. 차선 카테고리를 문서에 기록했는가?

## 20. 범위 변경과 사용자 피드백 처리

제작 중 사용자가 방향을 바꾸면 최신 요청을 기준으로 범위를 다시 고정한다.

- `이미지 말고 인포그래픽`: 사진 생성을 중단하고 HTML/CSS 정보 그래픽으로 전환한다.
- `커버 이미지만 변경`: hero, preload, OG, schema image, alt/caption만 바꾸고 본문 이미지는 유지한다.
- `콘텐츠 제작까지만`: 계정 로그인, 결제, 제출을 하지 않는다.
- `라이브 반영`: commit/push만으로 끝내지 않고 실제 도메인 응답을 확인한다.
- `링크 확인`: 소스뿐 아니라 canonical 페이지의 최종 href와 HTTP 상태를 확인한다.

버려진 생성 이미지는 사이트 자산으로 복사하지 않는다. 프로젝트에 참조되지 않은 파일이 남지 않도록 `git status`와 전체 텍스트 검색으로 확인한다.

## 21. 권장 문서 구조

프로젝트마다 다음 세 파일 역할을 분리한다.

```text
docs/quuu-promote-universal-playbook.md
    여러 사이트에서 재사용하는 절차와 기준

docs/quuu-promote-[article-slug].md
    개별 원고, 메타데이터, 공유 문구, 제출 상태

docs/current-status.md
    현재 라이브 여부, 커밋, 검증 결과, 다음 작업
```

개별 작업이 끝나면 최소한 아래를 남긴다.

- canonical 라이브 URL
- 최종 카테고리와 차선
- 최종 제목과 공유 문구
- 이미지/OG 경로
- 내부 링크와 외부 출처 수
- sitemap/robots/schema 상태
- 검증한 날짜와 커밋
- `Prepared`, `Published`, `Submitted` 중 정확한 상태
